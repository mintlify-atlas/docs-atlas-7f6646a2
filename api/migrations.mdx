This guide covers how to work with database migrations in the Hack Western Platform using Drizzle Kit.

## Overview

The platform uses [Drizzle ORM](https://orm.drizzle.team) for database management and [Drizzle Kit](https://orm.drizzle.team/kit-docs/overview) for migrations. Migrations are automatically generated by comparing your schema definition with the current database state.

## Configuration

Migration settings are defined in `drizzle.config.ts`:

```typescript
import { defineConfig } from "drizzle-kit";
import { env } from "~/env";

export default defineConfig({
  out: "./drizzle",
  schema: "./src/server/db/schema.ts",
  dialect: "postgresql",
  dbCredentials: {
    url: env.DATABASE_URL,
  },
  tablesFilter: ["hw_*"],
});
```

### Configuration options

- `out`: Directory where migration files are generated (`./drizzle`)
- `schema`: Path to your schema definition (`./src/server/db/schema.ts`)
- `dialect`: Database type (`postgresql`)
- `dbCredentials.url`: Database connection string from environment variables
- `tablesFilter`: Only manage tables matching this pattern (`hw_*`)

## Available commands

The following npm scripts are available in `package.json` for database operations:

```json
{
  "scripts": {
    "db:start": "bash ./start-database.sh",
    "db:studio": "drizzle-kit studio",
    "db:generate": "drizzle-kit generate",
    "db:migrate": "tsx ./src/server/db/migrate.ts",
    "db:seed": "dotenv tsx ./src/server/db/seed/index.ts"
  }
}
```

### db:start

Starts a local PostgreSQL database using Docker (development only).

```bash
npm run db:start
```

### db:studio

Launches Drizzle Studio, a web-based database browser for viewing and editing data.

```bash
npm run db:studio
```

This will open a browser at `https://local.drizzle.studio` where you can:
- Browse all tables and data
- Edit records directly
- Run queries
- View relationships

### db:generate

Generates a new migration file based on schema changes.

```bash
npm run db:generate
```

This command:
1. Compares your schema definition in `src/server/db/schema.ts` with the current database state
2. Generates SQL migration files in the `./drizzle` directory
3. Creates both a `.sql` file and a `.json` snapshot
4. Updates `drizzle/meta/_journal.json` with the migration metadata

**Example workflow:**

1. Modify your schema in `src/server/db/schema.ts`:

```typescript
export const users = createTable("user", {
  id: varchar("id", { length: 255 }).notNull().primaryKey(),
  name: varchar("name", { length: 255 }),
  email: varchar("email", { length: 255 }).notNull(),
  // Add a new column:
  bio: text("bio"),
});
```

2. Generate the migration:

```bash
npm run db:generate
```

3. Review the generated SQL in `drizzle/XXXX_<name>.sql`

### db:migrate

Runs all pending migrations against the database.

```bash
npm run db:migrate
```

This executes the migration script at `src/server/db/migrate.ts`:

```typescript
import "dotenv/config";
import { migrate } from "drizzle-orm/postgres-js/migrator";
import { conn, db } from ".";

// This will run migrations on the database, skipping the ones already applied
await migrate(db, { migrationsFolder: "./drizzle" });

// Don't forget to close the connection, otherwise the script will hang
await conn.end();
```

**The migration runner will:**
- Read all `.sql` files from the `./drizzle` directory
- Check which migrations have already been applied
- Execute only the pending migrations in order
- Track applied migrations in a `__drizzle_migrations` table

### db:seed

Seeds the database with test data (development only).

```bash
npm run db:seed
```

## Migration workflow

### Standard workflow

1. **Make schema changes** in `src/server/db/schema.ts`
2. **Generate migration**: `npm run db:generate`
3. **Review the generated SQL** in `drizzle/XXXX_<name>.sql`
4. **Run migrations**: `npm run db:migrate`
5. **Verify changes** using `npm run db:studio`

### Example: Adding a new table

1. Define the table in `src/server/db/schema.ts`:

```typescript
export const announcements = createTable("announcement", {
  id: serial("id").primaryKey(),
  title: varchar("title", { length: 255 }).notNull(),
  content: text("content").notNull(),
  createdAt: timestamp("created_at", { mode: "date", precision: 3 })
    .defaultNow()
    .notNull(),
  publishedAt: timestamp("published_at", { mode: "date" }),
});
```

2. Generate the migration:

```bash
npm run db:generate
```

3. This creates a file like `drizzle/0018_awesome_phoenix.sql`:

```sql
CREATE TABLE "hw_announcement" (
  "id" serial PRIMARY KEY NOT NULL,
  "title" varchar(255) NOT NULL,
  "content" text NOT NULL,
  "created_at" timestamp (3) DEFAULT now() NOT NULL,
  "published_at" timestamp
);
```

4. Apply the migration:

```bash
npm run db:migrate
```

### Example: Modifying a column

1. Update the schema:

```typescript
export const users = createTable("user", {
  // Change email length from 255 to 320
  email: varchar("email", { length: 320 }).notNull(),
  // ...
});
```

2. Generate and review:

```bash
npm run db:generate
```

Generated SQL:

```sql
ALTER TABLE "hw_user" ALTER COLUMN "email" TYPE varchar(320);
```

3. Apply the migration:

```bash
npm run db:migrate
```

### Example: Adding a foreign key relationship

1. Define the relationship:

```typescript
export const posts = createTable("post", {
  id: serial("id").primaryKey(),
  authorId: varchar("author_id", { length: 255 })
    .notNull()
    .references(() => users.id),
  title: varchar("title", { length: 255 }).notNull(),
});

export const postsRelations = relations(posts, ({ one }) => ({
  author: one(users, {
    fields: [posts.authorId],
    references: [users.id],
  }),
}));
```

2. Generate the migration:

```bash
npm run db:generate
```

Generated SQL:

```sql
CREATE TABLE "hw_post" (
  "id" serial PRIMARY KEY NOT NULL,
  "author_id" varchar(255) NOT NULL,
  "title" varchar(255) NOT NULL
);
--> statement-breakpoint
ALTER TABLE "hw_post" ADD CONSTRAINT "hw_post_author_id_hw_user_id_fk" 
  FOREIGN KEY ("author_id") REFERENCES "public"."hw_user"("id") 
  ON DELETE no action ON UPDATE no action;
```

## Migration files

### File structure

Migrations are stored in the `./drizzle` directory:

```
drizzle/
├── meta/
│   ├── _journal.json          # Migration history
│   ├── 0000_snapshot.json     # Schema snapshot
│   └── ...
├── 0000_safe_quentin_quire.sql    # Initial migration
├── 0001_slow_stephen_strange.sql
├── 0002_lazy_starbolt.sql
└── ...
```

### Migration naming

Drizzle Kit generates migration files with the format:

```
<index>_<random_name>.sql
```

Example: `0017_early_trauma.sql`

- Index is sequential (0000, 0001, 0002...)
- Random names are generated from a word list
- Names are unique but not meaningful

### Migration journal

The `drizzle/meta/_journal.json` file tracks all migrations:

```json
{
  "version": "7",
  "dialect": "postgresql",
  "entries": [
    {
      "idx": 0,
      "version": "6",
      "when": 1716846068647,
      "tag": "0000_safe_quentin_quire",
      "breakpoints": true
    },
    {
      "idx": 1,
      "version": "6",
      "when": 1717446092846,
      "tag": "0001_slow_stephen_strange",
      "breakpoints": true
    }
  ]
}
```

## Best practices

### Review before applying

Always review generated migrations before running them:

1. Check the SQL in `drizzle/XXXX_<name>.sql`
2. Verify it matches your intended changes
3. Look for potential data loss (dropped columns, changed types)
4. Test on a development database first

### Backup before production migrations

Before running migrations in production:

```bash
# Create a database backup
pg_dump $DATABASE_URL > backup.sql

# Run migrations
npm run db:migrate
```

### Handle data migrations

For complex schema changes that require data transformation, you may need to:

1. Generate the schema migration
2. Manually edit the `.sql` file to include data transformation
3. Or create a separate data migration script

Example - renaming a column with data preservation:

```sql
-- Generated by Drizzle
ALTER TABLE "hw_application" ADD COLUMN "country_of_residence" "country";

-- Manual data migration
UPDATE "hw_application" 
SET "country_of_residence" = "old_country_field"::"country"
WHERE "old_country_field" IS NOT NULL;

-- Generated by Drizzle
ALTER TABLE "hw_application" DROP COLUMN "old_country_field";
```

### Version control

Always commit migration files to version control:

```bash
git add drizzle/
git commit -m "Add user bio field migration"
```

### Table filtering

The `tablesFilter` configuration prevents Drizzle from managing tables outside your prefix:

```typescript
tablesFilter: ["hw_*"]  // Only manage hw_* tables
```

This is useful when:
- Multiple projects share a database
- You have legacy tables to preserve
- You want to exclude system tables

## Troubleshooting

### Migration hangs or times out

Check that `migrate.ts` closes the connection:

```typescript
await migrate(db, { migrationsFolder: "./drizzle" });
await conn.end();  // Important!
```

### "Migration already applied" errors

Drizzle tracks migrations in the `__drizzle_migrations` table. If this table gets out of sync:

1. Check the table contents:

```sql
SELECT * FROM __drizzle_migrations;
```

2. Manually add or remove entries if needed

### Schema drift

If your database schema doesn't match your code:

1. Use Drizzle Studio to inspect the current state
2. Generate a new migration to sync
3. Or reset the database (development only):

```bash
# Drop all tables
psql $DATABASE_URL -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"

# Run all migrations from scratch
npm run db:migrate
```

### Connection issues

Verify your `DATABASE_URL` environment variable:

```bash
echo $DATABASE_URL
# Should output: postgresql://user:password@host:port/database
```

## Production deployment

For production deployments, run migrations as part of your deployment pipeline:

```bash
# Example deployment script
npm install
npm run build
npm run db:migrate  # Run migrations before starting the app
npm start
```

Or use a separate migration job:

```yaml
# Example GitHub Actions
- name: Run database migrations
  run: npm run db:migrate
  env:
    DATABASE_URL: ${{ secrets.DATABASE_URL }}

- name: Deploy application
  run: npm run deploy
```
