This page documents the complete database schema for the Hack Western Platform, including all tables, columns, relationships, and enums.

## Table prefix

All tables use the prefix `hw_` (Hack Western). This prefix is configured in the schema and can be changed for different years.

```typescript
const TABLE_PREFIX = "hw";
export const createTable = pgTableCreator((name) => `${TABLE_PREFIX}_${name}`);
```

## Enums

### application_status

The status of a hacker application, from when it's first started to final decision.

- `IN_PROGRESS` - Application is being filled out
- `PENDING_REVIEW` - Submitted and waiting for review
- `IN_REVIEW` - Currently being reviewed by organizers
- `ACCEPTED` - Application accepted
- `REJECTED` - Application rejected
- `WAITLISTED` - Application waitlisted
- `DECLINED` - Applicant declined their acceptance

### user_type

The type of user account.

- `hacker` - Participant/applicant
- `organizer` - Event organizer with admin access
- `sponsor` - Sponsor representative

### avatar_colour

Avatar color options for hacker profiles.

- `red`
- `orange`
- `yellow`
- `green`
- `blue`
- `purple`
- `pink`

### level_of_study

The school/university year that the hacker applicant is in.

- `Less than Secondary / High School`
- `Secondary / High School`
- `Undergraduate University (2 year - community college etc.)`
- `Undergraduate University (3+ year)`
- `Graduate University (Masters, Professional, Doctoral, etc)`
- `Code School / Bootcamp`
- `Other Vocational / Trade Program or Apprenticeship`
- `Post Doctorate`
- `Other`
- `Not currently a student`
- `Prefer not to answer`

### major

- `Other Engineering Discipline`

## Tables

### users

Core user accounts table for all users (hackers, organizers, sponsors).

**Table name:** `hw_user`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | varchar(255) | PRIMARY KEY, NOT NULL | Unique user identifier |
| name | varchar(255) | | User's display name |
| password | varchar(255) | | Hashed password for credential auth |
| email | varchar(255) | NOT NULL | User's email address |
| emailVerified | timestamp | DEFAULT CURRENT_TIMESTAMP | Email verification timestamp |
| image | varchar(255) | | Profile image URL |
| type | user_type | DEFAULT 'hacker' | User account type |
| scavengerHuntEarned | integer | DEFAULT 0 | Total points earned in scavenger hunt |
| scavengerHuntBalance | integer | DEFAULT 0 | Current point balance for redemptions |

**Indexes:**
- `user_scavenger_hunt_earned_idx` on `scavengerHuntEarned`
- `user_scavenger_hunt_balance_idx` on `scavengerHuntBalance`

**Relationships:**
- One-to-many with `accounts` (NextAuth OAuth accounts)
- One-to-one with `applications`

### reviews

Organizer reviews of hacker applications for the review portal.

**Table name:** `hw_review`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| reviewerUserId | varchar(255) | PRIMARY KEY (with applicantUserId), NOT NULL, FK → users.id | Organizer conducting the review |
| applicantUserId | varchar(255) | PRIMARY KEY (with reviewerUserId), NOT NULL, FK → users.id, FK → applications.userId | Applicant being reviewed |
| createdAt | timestamp(3) | NOT NULL, DEFAULT now() | Review creation timestamp |
| updatedAt | timestamp(3) | NOT NULL, DEFAULT now() | Last update timestamp |
| originalityRating | smallint | DEFAULT 0 | Originality score |
| technicalityRating | smallint | DEFAULT 0 | Technical skills score |
| passionRating | smallint | DEFAULT 0 | Passion/enthusiasm score |
| comments | text | | Reviewer's comments |
| completed | boolean | DEFAULT false | Review completion status |
| referral | boolean | DEFAULT false | Marked for referral/special consideration |

**Composite primary key:** (`reviewerUserId`, `applicantUserId`)

**Indexes:**
- `applicant_user_id_idx` on `applicantUserId`

**Relationships:**
- Many-to-one with `users` (reviewer)
- Many-to-one with `users` (applicant)
- Many-to-one with `applications`

## Relationships overview

### User-centric relationships

- **users** → **accounts**: One-to-many (a user can have multiple OAuth providers)
- **users** → **sessions**: One-to-many (a user can have multiple active sessions)
- **users** → **applications**: One-to-one (each user has one application)
- **users** → **reviews** (as reviewer): One-to-many (organizers review many applications)
- **users** → **reviews** (as applicant): One-to-many (applications receive many reviews)

### Application review workflow

- **applications** → **users**: Many-to-one (application belongs to one user)
- **applications** → **reviews**: One-to-many (application can have multiple reviews)
- **reviews** → **users** (reviewer): Many-to-one (review by one organizer)
- **reviews** → **users** (applicant): Many-to-one (review for one applicant)
- **reviews** → **applications**: Many-to-one (review for one application)

### Scavenger hunt relationships

- **scavengerHuntScans** → **users** (userId): Many-to-one (scan by one user)
- **scavengerHuntScans** → **users** (scannerId): Many-to-one (facilitated by one organizer)
- **scavengerHuntScans** → **scavengerHuntItems**: Many-to-one (scan of one item)
- **scavengerHuntRedemptions** → **users**: Many-to-one (redemption by one user)
- **scavengerHuntRedemptions** → **scavengerHuntRewards**: Many-to-one (redemption of one reward)
